# 页面采集待解决问题

## 1. 监控信息页图表数值暂无法结构化采集

### 当前现象

- 开发机 / worker 的 `监控信息` 页可以识别当前子页
- 可以拿到图表标题、图例、坐标轴等可见文字
- 但图表里的真实时序数据点目前无法稳定写入 JSON

### 已确认情况

- 页面图表区域存在 `canvas` / 图表渲染痕迹
- 当前方案读取 DOM 文本时，只能拿到：
  - 图表标题
  - 图例
  - 坐标轴刻度
- 不能稳定拿到：
  - 每个时间点的 CPU / 内存 / GPU 数值
  - 曲线背后的原始 series 数据

### 暂定结论

- `监控信息` 页先只识别页面和子页，不将图表数值纳入生产 JSON
- 后续优先尝试：
  1. 查找图表接口 / Prometheus / query_range 请求
  2. 查找前端图表实例是否暴露 series 数据
  3. 最后才考虑 OCR/截图兜底

## 2. Terminal 页正文暂无法稳定采集

### 当前现象

- Terminal 页能够识别出来
- 但终端窗口里可见的命令行文本目前无法稳定写入 JSON

### 已确认情况

根据蓝本插件探测：

- `hasXtermLikeNode = true`
- `hasCanvas = true`
- `panelTextLength = 0`
- `activePanelTextSample = ""`

说明：

- 当前 Terminal 大概率由 `xterm.js` + `canvas` 渲染
- 可见终端内容没有以普通 DOM 文本暴露出来
- 所以视觉上能看到内容，不代表插件能通过 `innerText` 读取到

### 暂定结论

- Terminal 页目前只记录：
  - 当前 `activeTab = terminal`
  - `terminalInfo` 可作为占位信息
- 不将终端正文纳入生产 JSON
- 后续优先尝试：
  1. 查找 xterm 可访问文本层 / 辅助层
  2. 查找 websocket / terminal proxy 接口
  3. 最后再考虑 OCR 兜底

## 3. RJob 日志页目前只能抓可见正文，暂未稳定结构化解析

### 当前现象

- `日志` 子页可以抓到页面里可见的黑底日志正文
- 但日志中的训练指标、step、loss、acc、eta 等内容，目前还是整段文本
- 还没有稳定拆成可直接消费的结构化字段

### 已确认情况

- 蓝图插件已经验证日志正文暴露在 DOM 中
- 当前主要命中容器：
  - `.log_wrapper`
  - `.replica_log_content`
- 所以“抓到日志原文”是可以做到的
- 但以下能力目前仍不稳定：
  - 自动按行拆分为日志记录
  - 自动从每行提取 `timestamp / step / epoch / loss / acc / eta` 等字段
  - 区分训练日志、报错日志、系统提示日志

### 暂定结论

- `日志` 子页当前先以“原始日志正文字符串”为主
- 暂不把训练指标解析作为生产版稳定能力
- 后续优先尝试：
  1. 设计日志行切分规则
  2. 为常见训练日志补正则解析
  3. 再决定是否输出结构化 metrics 数组
