# 数据采集层

## 定位

数据采集层只做一件事：

- 把业务页面中的信息抓出来，并整理成统一的结构化 JSON

它不负责：

- LLM 分析
- 业务结论判断
- 后端存储

可以把整个链路理解成：

- 数据采集层：抓页面、识别结构、输出 JSON
- 分析层：基于 JSON 做诊断和总结
- 消费层：存储、检索、编排、接口输出

## 输入与输出

输入：

- 当前页面 DOM
- 页面可见文本
- 表格
- 日志区
- YAML 区域

输出：

- 统一 schema 的 JSON

目标不是“把页面截图保存下来”，而是“让页面内容可被程序和模型稳定理解”。

## 核心流程

数据采集层的处理流程固定为 5 步：

1. 页面识别
   判断当前是 `开发机`、`worker`、`rjob` 还是其他页面。

2. 子页识别
   判断当前是 `状态日志`、`yaml`、`日志`、`进程`、`镜像` 等哪个子页。

3. 数据区域定位
   判断信息主要落在哪种结构里：
   - 键值对
   - 表格
   - 文本块
   - 日志区
   - YAML 区

4. 抽取策略选择
   按区域选择对应抽取方式：
   - 键值对抽取
   - 表格抽取
   - 原文保留
   - 蓝本探测

5. 标准化输出
   统一整理成固定 JSON，而不是每个页面各自输出一套格式。

## 当前结构设计

当前 JSON 设计遵循几个原则：

- 先识别页面，再抽字段
- 基础信息和当前子页信息分开
- 结构化优先，原文兜底
- 英文 `key` 和中文 `label` 同时保留
- 表格按“记录”组织，不按纯位置数组组织

一个简化后的结构可以理解成：

```json
{
  "captureMode": "production",
  "entityType": "rjob",
  "activeTab": "statusLog",
  "standardized": [],
  "statusLogInfo": {}
}
```

其中：

- `standardized`：放稳定基础信息
- 顶层并行字段：放当前子页内容，比如 `statusLogInfo`、`yamlInfo`、`jobLogInfo`

## 两类采集模式

当前有两种模式：

### 1. 生产采集

面向稳定输出。

特点：

- schema 固定
- 调试信息少
- 适合后端正式消费

对应目录：

- `diagnosis-data-collector`

### 2. 蓝本采集

面向页面探索。

特点：

- 输出更多页面结构信息
- 用于定位表格、文本块、日志区、YAML 区
- 适合新页面接入前摸底

对应目录：

- `page-blueprint-capture`

## 扩展方法

以后接入新页面时，统一按下面方式处理：

1. 先确认能否识别页面类型
2. 再确认能否识别当前子页
3. 判断主要数据落在哪种结构里
4. 先抽稳定字段
5. 抽不稳的内容先保留原文
6. 最后统一放入标准 JSON

如果页面结构不清楚，先用蓝本采集，再决定生产版怎么补规则。

## 当前价值

这层的价值不只是“抓数据”，而是把复杂页面整理成统一的数据入口。

这样做的好处是：

- 后端拿到的数据格式更稳定
- LLM 不需要直接面对杂乱页面
- 新页面接入时有统一方法
- 后续扩展不同页面时不容易失控
