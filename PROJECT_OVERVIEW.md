# 智能诊断助手项目说明

## 项目目标

这个项目的目标是为内部任务诊断场景开发一套“浏览器插件 + 后端服务”的采集方案。

当前阶段重点是先把浏览器侧页面信息稳定抓出来，形成可供后端和后续 LLM 使用的 JSON 数据。

核心思路：

- 用 Chrome 插件进入用户当前已登录的业务页面
- 识别当前页面类型和当前子页
- 抓取页面中可见且稳定的关键信息
- 输出标准化 JSON，供后端继续处理、诊断和总结

## 当前目录说明

项目当前包含 2 个主要插件目录：

### 1. `page-blueprint-capture`

页面蓝本采集器。

作用：

- 用来探索页面结构
- 输出页面蓝本信息，帮助判断某个页面的 DOM、文本块、表格、日志区域、yaml 区域到底在哪里
- 适合新页面、新子页、新异常场景排查

特点：

- 输出 `captureMode: "blueprint"`
- 输出 `blueprint` 调试信息
- 更适合“看页面长什么样、信息落在哪”

### 2. `diagnosis-data-collector`

诊断数据采集器。

作用：

- 面向后端正式消费
- 输出标准化 JSON
- 尽量减少调试噪音

特点：

- 输出 `captureMode: "production"`
- 不输出蓝图结构
- 重点是稳定、固定 schema、方便后端调用

## 当前 JSON 设计原则

### 1. 基础信息和当前子页信息分开

当前约定：

- `standardized` 里只放稳定基础信息
- 当前子页内容放在顶层并行字段

例如：

- `statusLogInfo`
- `jobLogInfo`
- `yamlInfo`
- `processInfo`

### 2. 字段同时保留英文 key 和中文 label

每个字段当前使用统一结构：

```json
{
  "key": "status",
  "label": "状态",
  "value": "运行中"
}
```

这样做的目的：

- 后端逻辑可以使用英文 key
- 展示和交互时可以直接用中文 label

### 3. 表格信息按“记录”组织

对于状态日志、进程等表格型内容，当前逐步统一为“每条记录一组字段”的形式，方便后端和 LLM 直接理解，不再依赖纯数组位置对应。

## 当前已知限制

当前已知限制已单独记录在：

[KNOWN_LIMITATIONS.md](/Users/che/projects/html/KNOWN_LIMITATIONS.md)

目前主要包括：

### 1. 监控信息

- 可以识别页面和子页
- 但图表中的真实时序数值还无法稳定结构化采集

### 2. Terminal

- 可以识别 Terminal 子页
- 但由于页面大概率使用 `xterm.js + canvas` 渲染，终端正文无法稳定读取

### 3. RJob 日志细粒度结构化

- 可以抓到可见日志正文
- 但 `step / loss / acc / eta / epoch` 等字段还没有稳定结构化输出

## 当前项目进展总结

到目前为止，这个项目已经完成了“从页面中稳定拿到基础诊断信息”的第一阶段工作。

已经具备的能力：

- 区分页面类型
- 区分当前子页
- 输出统一 JSON
- 抓取开发机 / worker / rjob 三类页面的核心基础信息
- 抓取 yaml、状态日志、部分日志正文等高价值内容

还在继续完善的能力：

- 监控图表真实数值抓取
- Terminal 正文抓取
- RJob 日志结构化解析
- 生产版插件的进一步收敛和稳定化

## 下一步建议

建议后续按以下顺序继续推进：

1. 稳定 `diagnosis-data-collector`，减少临时分支和调试逻辑
2. 优先补强高价值页面：
   - `监控信息`
   - `RJob 日志结构化`
3. 再考虑把采集结果发送到本地后端
4. 最后再进入真正的诊断 agent 阶段

## 当前结论

这个项目现在已经不是“能不能抓页面”的问题，而是已经进入“哪些页面可以稳定标准化、哪些页面需要继续攻克”的阶段。

也就是说：

- 页面采集方向已经验证可行
- 蓝图插件已经证明很多页面的正文是可抓的
- 生产版插件已经具备基本标准化能力
- 下一阶段重点是稳定性和结构化深度
